\section{Introduction and Motivation}\label{sec:intro}
% ══════════════════════════════════════════════════════════════════════════════

Modern recommendation systems generate personalised suggestions by comparing a
target user's preferences with the preferences of many other users. This project
focuses on that problem: as the number of users and items grows, the similarity
calculation becomes too slow for a simple serial program. Pearson correlation is
used because it measures how closely two users' rating patterns move together.
After the system finds similar users, it predicts an unseen rating by combining
the neighbours' ratings with weights based on their similarity.

The main cost appears during user similarity computation. For the benchmark data
used in this report, there are $1{,}000$ users and $1{,}000$ items. Each user
must be compared with all other users, and each comparison may scan many item
ratings to compute one Pearson similarity value. That creates hundreds of
thousands of user pair comparisons and roughly hundreds of millions of rating
checks before prediction begins. This is the exact situation described in the
proposal: large similarity computations are useful for recommendation systems,
but they become slow as the data size increases. Therefore the Pearson
similarity phase is the main bottleneck and the main target for parallelisation.

This report documents a \textbf{Pearson correlation based recommender}
implemented in seven versions: serial, OpenMP, POSIX threads, MPI, CUDA, hybrid
MPI+OpenMP, and hybrid MPI+Pthreads. All versions use the same synthetic
ratings matrix ($1{,}000$ users $\times\ 1{,}000$ items, \code{SEED=42}, 70\%
sparse, TOP K $= 20$), so the report compares only the effect of the
parallel programming model. The evaluation checks two questions: whether each
parallel version preserves the serial result, and how much runtime improves
as workers increase from 1 to 8. Every parallel version reproduces the serial
result exactly. For this implementation-correctness evaluation, MAE and RMSE
are measured as the average error of each implementation's prediction array
against the serial prediction array. Therefore the serial baseline and every
parallel version report MAE $=0.0000$ and RMSE $=0.0000$; the similarity-matrix
checksum is bit-identical at
$942.387323$, confirming race free partitioning. CUDA is fastest at $38.78\times$ over serial; OpenMP
leads the CPU versions at $6.26\times$ on eight threads.

\vspace{0.6em}
\noindent\textbf{Platform.} Pop!\_OS 24.04 LTS $\cdot$ GCC 13.3.0 $\cdot$ Open MPI
4.1.6 $\cdot$ Intel Core i7 11800H (8 physical / 16 logical
cores) $\cdot$ NVIDIA GeForce RTX 3050 Laptop GPU.

\section{Algorithm Design and Parallel Programming Concepts}\label{sec:design}
% ══════════════════════════════════════════════════════════════════════════════

\subsection{Algorithm Pipeline (Five Phases)}

Every version executes the same five sequential phases summarised in
Table~\ref{tab:phases}. Phases~3 and~4 dominate runtime and are the main
parallelisation targets; Phase~2 is also row parallel in the parallel versions.
Data generation stays serial in all versions because C's \code{rand()} is not
thread safe, and running the same \code{srand(42)} sequence everywhere
guarantees a byte identical input matrix without any communication.

\begin{table}[!htbp]
\centering
\begin{tabular}{C{1.2cm} L{3cm} L{6.5cm} L{3.5cm}}
\toprule
\textbf{Phase} & \textbf{Name} & \textbf{Description} & \textbf{Complexity} \\
\midrule
1 & Data Generation  & Synthetic sparse matrix; 10\% test split & $O(N \times M)$ \\
2 & User Means       & Mean for each user $\mu_u$              & $O(N \times M)$ \\
3 & Similarity       & Pearson for all $(u,v)$ pairs, the bottleneck & $O(N^2 \times M)$ \\
4 & Predictions      & TOP K weighted prediction               & $O(N^2 \times M)$ \\
5 & Correctness Evaluation & MAE/RMSE against serial predictions & $O(|T|)$ \\
\bottomrule
\end{tabular}
\caption{Algorithm phases and computational complexity.}
\label{tab:phases}
\end{table}

The two heavy phases use the following formulas. Pearson similarity (Phase~3):
\[
  \text{sim}(u,v) = \frac{\sum_i (R_{ui}-\mu_u)(R_{vi}-\mu_v)}
    {\sqrt{\sum_i (R_{ui}-\mu_u)^2}\;\sqrt{\sum_i (R_{vi}-\mu_v)^2}}
\]
and the TOP K weighted prediction (Phase~4):
\[
  \hat{r}_{ui} = \mu_u + \frac{\sum_{k \in N_K(u)} \text{sim}(u,k)\,(R_{ki}-\mu_k)}
                              {\sum_{k \in N_K(u)} \text{sim}(u,k)}.
\]
Both are \emph{embarrassingly parallel at the row level}: the work for one user
row is independent of every other, which is what makes the same algorithm map
cleanly onto all five parallel models. Figure~\ref{fig:pipeline} shows the
pipeline and marks which phases each technology parallelises; the row wise
decomposition that keeps every version race free is then detailed in the
per technology diagrams that follow.

\begin{figure}[!htbp]
  \centering
  \includegraphics[width=\linewidth]{../analysis_diagrams/drawio/png/diagram_01_algorithm_pipeline.png}
  \caption{The five phase recommender pipeline, shared by every implementation.
  The same algorithm and data (\code{SEED=42}) are used throughout; only the
  runtime changes. Phase~3 (Pearson similarity) is the $O(N^2 M)$ bottleneck;
  Phases~3--4 are the main targets of all five parallel models, Phase~2 is also
  row parallel in the parallel implementations, and data generation stays
  serial.}
  \label{fig:pipeline}
\end{figure}

\subsection{OpenMP: Shared Memory Fork Join Parallelism}

OpenMP exploits shared memory parallelism through compiler directives: a master
thread forks into $T$ workers that share one address space, then rejoins at an
implicit barrier (Figure~\ref{fig:openmp}). The decisive design choice is in
Phase~3, where the outer loop carries \code{\#pragma omp parallel for
schedule(dynamic,4)}. Dynamic scheduling is essential because the triangular
similarity loop is badly imbalanced; the first row performs $N{-}1 = 999$
Pearson calls while the last performs only one. Dynamic scheduling lets idle
threads pull four row chunks from a shared work queue, so no core sits idle.

The version stays lock free without any mutex: each thread owns an exclusive set
of outer loop rows $u$ and writes \code{SIM(u,v)} only for those rows, so no two
threads ever touch the same cell. In Phase~5 the \code{reduction(+:err)} clause
gives each thread a private accumulator that OpenMP sums at the barrier, and the
same pattern (\code{reduction(+:sq)}) accumulates the squared prediction
difference for RMSE.

\begin{figure}[!htbp]
  \centering
  \includegraphics[width=\linewidth]{../analysis_diagrams/drawio/png/diagram_02_openmp.png}
  \caption{OpenMP fork/join model. The master thread forks a team of $T$ workers
  over one shared address space (\code{ratings}, \code{sim\_matrix}, \ldots), each
  owning a disjoint slice of user rows, then rejoins at an implicit barrier. The
  side notes show the three race free techniques used: \code{schedule(dynamic,4)}
  for the triangular loop, disjoint row writes, and a thread private \code{nbrs[]}
  buffer with \code{reduction(+:err)}.}
  \label{fig:openmp}
\end{figure}

\subsection{Pthreads: Manual POSIX Thread Management}

POSIX threads require the programmer to create threads, assign work, and
synchronise explicitly (Figure~\ref{fig:pthreads}). A \code{run\_parallel(fn,
nthreads)} harness is invoked at the start of Phases~2, 3, and~4: it calls
\code{pthread\_create()} to spawn $T$ threads, each receiving a
\code{ThreadArgs} struct with its \code{tid} and the total \code{nthreads},
lets every thread derive its exclusive row range
$\text{start} = tid \times \lceil N/T\rceil$,
$\text{end} = \min(\text{start}+\lceil N/T\rceil,\,N)$, and then uses
\code{pthread\_join()} as a phase barrier.

The key difference from OpenMP is twofold: threads are created and joined three
separate times (once per phase), paying OS thread creation overhead three times
per run, and the static ceiling division row assignment cannot compensate for the
triangular loop imbalance. Both effects explain the slightly lower efficiency
observed against OpenMP in Section~\ref{sec:timing}.

\begin{figure}[!htbp]
  \centering
  \includegraphics[width=\linewidth]{../analysis_diagrams/drawio/png/diagram_03_pthreads.png}
  \caption{Manual POSIX thread management. \code{main()} calls \code{run\_parallel}
  once per parallel phase: \code{pthread\_create} spawns $T$ threads, each deriving
  its own \code{ceil} division row range from a \code{ThreadArgs} struct, and
  \code{pthread\_join} on all threads acts as the phase barrier. Disjoint rows plus
  a private \code{nbrs[]} buffer make it lock free.}
  \label{fig:pthreads}
\end{figure}

\subsection{MPI: Distributed Memory Parallelism}

Each MPI rank is an independent process with its own private address space
(Figure~\ref{fig:mpi}). The $N=1{,}000$ users are divided evenly: rank $r$ owns
rows $[r \cdot N/P,\; (r{+}1)\cdot N/P)$. Because the ranks cannot share memory,
data has to be exchanged explicitly through collective operations. Concretely:
Phase~1 needs no communication (all ranks call \code{srand(42)} independently);
Phase~2 computes a local \code{user\_mean[]} slice and gathers it with
\code{MPI\_Allgatherv}; Phase~3 fills the rank's rows of the similarity matrix
and rebuilds the full matrix on every rank, again with \code{MPI\_Allgatherv};
Phase~4 predicts the rank's own users with no communication; and Phase~5 sums the
local serial-vs-parallel prediction error to rank~0 via \code{MPI\_Reduce(SUM)}
for the global MAE/RMSE.

The \code{MPI\_Allgatherv} that rebuilds the $N \times N$ similarity matrix
transfers $\approx 4$\,MB at $N=1{,}000$. This is the cost that limits
single node efficiency, but it is also exactly the mechanism that makes the code
work unchanged across multiple physical nodes, where shared memory is not an
option.

\begin{figure}[!htbp]
  \centering
  \includegraphics[width=\linewidth]{../analysis_diagrams/drawio/png/diagram_04_mpi.png}
  \caption{MPI distributed memory layout. Each rank is a separate process with a
  private copy of the matrices, computing the similarity rows it owns. A single
  \code{MPI\_Allgatherv} then gives every rank the complete $N\times N$ matrix
  (so predictions need no communication), and \code{MPI\_Reduce} sums the partial
  serial-vs-parallel prediction error to rank~0 for the global MAE/RMSE.}
  \label{fig:mpi}
\end{figure}

\subsection{CUDA: GPU Massively Parallel Computing}

The CUDA version (\code{cuda/cuda\_recommender.cu}) is fully implemented and
benchmarked on an NVIDIA GeForce RTX 3050 Laptop GPU (Ampere, compute capability
8.6, 16 SMs); measured results appear in Section~\ref{sec:cuda-results}. It maps
the embarrassingly parallel structure of the problem onto one GPU thread per
output element, using three kernels: \code{kernel\_user\_means} (one thread per
user, launched as \code{<<<(N+255)/256,\,256>>>}), \code{kernel\_similarity}
(one thread per $(u,v)$ pair on a 2D grid of $16\times16$ thread blocks), and
\code{kernel\_predictions} (one thread per $(u,i)$ cell on a 2D users\,$\times$\,items grid).

The similarity kernel alone launches a grid with
$63 \times 63 \times 256 = 1{,}016{,}064$ logical threads
(Figure~\ref{fig:cuda}). Because device code cannot call
\code{qsort}, each prediction thread keeps its TOP K neighbours in a small
fixed size register buffer instead. Timing uses \code{cudaEventRecord} so that
the reported kernel time excludes the \code{cudaMemcpy} transfer overhead.

\begin{figure}[!htbp]
  \centering
  \includegraphics[width=\linewidth]{../analysis_diagrams/drawio/png/diagram_05_cuda.png}
  \caption{CUDA GPU model. After a Host$\rightarrow$Device copy of the ratings,
  three kernels run in sequence on a 2D grid of $16\times16$ thread blocks, with
  one thread per $(u,v)$ pair / $(\text{user},\text{item})$ cell, before the
  predictions are copied back. The left column shows the GPU memory hierarchy
  (registers $\rightarrow$ shared $\rightarrow$ global $\rightarrow$ host).
  Benchmarked on an NVIDIA RTX 3050 (compute capability 8.6).}
  \label{fig:cuda}
\end{figure}

\subsection{Hybrid MPI+OpenMP: Two Level Parallelism}

The hybrid version combines coarse grain MPI (across nodes) with fine grain
OpenMP (within a node), as shown in Figure~\ref{fig:hybrid}. With $P$ MPI ranks
and $T$ OpenMP threads per rank, the total worker count is $P \times T$ while
only $P$ messages are sent per collective call. The
\code{MPI\_THREAD\_FUNNELED} mode (selected via \code{MPI\_Init\_thread}) guarantees
that every MPI call is made from the main thread only, after the OpenMP parallel
regions have closed. Four layouts are tested at a fixed eight total workers:
$2\times4$ (fewer MPI messages, more shared memory parallelism),
$4\times2$ (balanced communication vs.\ threading overhead),
$8\times1$ (equivalent to pure MPI), and
$1\times8$ (pure OpenMP wrapped with MPI overhead).

\begin{figure}[!htbp]
  \centering
  \includegraphics[width=\linewidth]{../analysis_diagrams/drawio/png/diagram_06_hybrid_mpi_openmp.png}
  \caption{Hybrid two level parallelism. Level~1 (MPI) partitions user rows across
  $P$ ranks on separate nodes; level~2 (OpenMP) spawns $T$ threads over each rank's
  rows in shared memory, for $P\times T$ total workers. \code{MPI\_Allgatherv} /
  \code{MPI\_Reduce} link the ranks, and \code{MPI\_THREAD\_FUNNELED} keeps all MPI
  calls on the main thread.}
  \label{fig:hybrid}
\end{figure}

\subsection{Hybrid MPI+Pthreads: Two Level Parallelism, POSIX Inner Layer}

A second hybrid variant keeps the outer MPI layer unchanged and replaces the
inner OpenMP layer with raw POSIX threads (Figure~\ref{fig:hybrid_pt}). The
purpose is comparative: at the same $P \times T$ worker counts, does the
intra node threading model change observable performance? The outer structure
is byte identical to Section~2.6, with the same \code{get\_range} row partition,
same \code{MPI\_THREAD\_FUNNELED} contract, same \code{MPI\_Allgatherv} after
Phases~2 and~3, and the same \code{MPI\_Reduce} for the global prediction
error. Only the inner loop differs: a \code{run\_parallel(fn,
threads, args, start, end)} harness spawns $T$ pthreads per phase, each
carrying its rank's row window in its \code{ThreadArgs} and deriving its
sub range via the same \code{work\_range} ceiling division used by the pure
pthreads version. Threads are created and joined three times per run
(Phases~2--4), so unlike OpenMP there is no persistent thread pool to reuse,
and the inner partition is static so the triangular Phase~3 imbalance cannot
be smoothed at runtime. The same four $P\times T$ layouts are tested as in the
MPI+OpenMP hybrid ($2\times4$, $4\times2$, $8\times1$, $1\times8$, all eight
total workers), so the two variants can be compared head to head; at
$8\times1$ this reduces to pure MPI (one pthread per rank), and at $1\times8$
to pure pthreads wrapped with MPI overhead.

\paragraph{Fairness controls.} The comparison is kept fair by construction:
\begin{itemize}[noitemsep,topsep=2pt]
  \item The same \code{SEED=42} gives every version a byte identical ratings
        matrix and test split, since Phase~1 runs the same serial \code{rand()}
        sequence everywhere.
  \item The same fixed parameters ($N=M=1{,}000$, 70\% sparsity, TOP K $=20$)
        are compiled into every source file.
  \item Reported timing covers only the similarity and prediction phases:
        the \code{[Timing] Total (sim+pred)} line that each binary prints,
        excluding data generation and evaluation.
\end{itemize}

\paragraph{Measurement protocol.} Speedup is computed as
$S(p) = T_{\text{serial}} / T_{\text{parallel}}(p)$ from the total
similarity plus prediction time. Thread and process counts are swept over
$\{1,2,4,8\}$; both hybrid variants (MPI+OpenMP and MPI+Pthreads) are measured
at the four $P\times T$ layouts ($2\times4$, $4\times2$, $8\times1$,
$1\times8$, all at eight total workers); and CUDA is benchmarked once on the GPU. Correctness is
reported with MAE/RMSE against the serial prediction array and the
\code{[Check]} similarity checksum emitted by every run.

\section{Correctness Analysis: Parallel vs.\ Serial}\label{sec:accuracy}
% ══════════════════════════════════════════════════════════════════════════════

For implementation correctness, the expected value is the serial prediction and
the observed value is the prediction produced by each parallel implementation
for the same $(u,i)$ pair. This interpretation is used throughout the report:

\[
  \text{MAE} = \frac{1}{|T|} \sum_{(u,i) \in T}
        \bigl|\hat{r}^{\text{par}}_{ui} - \hat{r}^{\text{ser}}_{ui}\bigr|,
  \quad
  \text{RMSE} = \sqrt{\frac{1}{|T|} \sum_{(u,i) \in T}
        \bigl(\hat{r}^{\text{par}}_{ui} - \hat{r}^{\text{ser}}_{ui}\bigr)^2}.
\]
If every implementation reproduces the serial prediction array exactly, both
metrics are $0.0000$. The serial row is also $0.0000$ because it is
the baseline compared against itself.

\label{tab:acc}
\end{table}


As Table~\ref{tab:acc} shows, all 22 runs produce MAE
$=\textbf{0.0000}$, RMSE $=\textbf{0.0000}$, and checksum
$=\textbf{942.387323}$, confirming zero race conditions and correct
partitioning across every implementation. Crucially, the CUDA version, despite
using completely different floating point hardware and a register based TOP K
selection rather than the CPU's \code{qsort}, reproduces the serial result
exactly. The MPI+Pthreads hybrid also matches exactly, confirming that swapping
the intra node threading model from OpenMP to raw pthreads is purely a
\emph{runtime} change and does not perturb the algorithm or partitioning.

\subsection{Performance Metric Definitions}

\[
  S(p) = \frac{T_{\text{serial}}}{T_{\text{parallel}}(p)}
  \qquad
  E(p) = \frac{S(p)}{p}
  \qquad
  S_{\max}(p) = \frac{1}{f + (1-f)/p}
\]

The measured speedups are interpreted with Amdahl's Law as an upper bound model,
not as a precise fitted decomposition. The benchmark total used below is
\code{similarity + prediction}, so setup, data generation, and evaluation are
outside the reported speedup denominator. The gap between the ideal eight worker
limit and the measured CPU results is therefore mainly explained by practical
overheads: scheduling, communication, memory bandwidth, and reduced all core
turbo clocks.

\subsection{OpenMP vs.\ Pthreads}

OpenMP ($\times$6.26, 78\% efficiency) outperforms Pthreads ($\times$5.74, 72\%)
for three reasons. First, \emph{dynamic load balancing}:
\code{schedule(dynamic,4)} adapts to the triangular loop imbalance, which the
static ceiling division in Pthreads cannot. Second, \emph{thread reuse}: OpenMP
reuses one thread team across Phases~2--4, whereas Pthreads creates and joins
threads three times per run. Third, \emph{runtime optimisations}: GCC's OpenMP
runtime applies platform specific scheduling heuristics. The two are closest at
low thread counts (within 1\% at $T=1$) and the gap widens with thread count, as
expected when the triangular imbalance has more workers to redistribute.

\subsection{MPI Scaling}

MPI efficiency declines from 80\% ($P=1$) to 64\% ($P=8$). The
\code{MPI\_Allgatherv} that rebuilds the $N \times N$ similarity matrix
($\approx 4$\,MB) limits single node performance. These results do not measure
multi node execution; they show that MPI is useful but carries communication
cost on the single node platform used here.

\subsection{Hybrid MPI+OpenMP: Inversion on a Single Node}

On a single node, hybrid performance rises sharply with the MPI rank count and
then flattens once the rank count matches the number of physical cores:
\begin{center}
Hybrid 4$\times$2 ($\times$4.84) $\approx$ 8$\times$1 ($\times$4.75)
$\gg$ 2$\times$4 ($\times$2.14) $>$ 1$\times$8 ($\times$1.09).
\end{center}
The 4$\times$2 and 8$\times$1 configurations are statistically tied at
$\approx$60\% efficiency. In this run 4$\times$2 edges 8$\times$1 by under
2\%, having more aggressive OMP scheduling absorb the triangular imbalance
within each rank's slice. Below that, the picture is unambiguous: when fewer
MPI ranks are used, the MPI startup and \code{MPI\_Allgatherv} cost dominates
what is essentially an OpenMP workload. The Hybrid 1$\times$8 configuration
pays all the MPI overhead while gaining zero distribution benefit.
\textbf{On this single node setup, the hybrid model does not outperform pure
OpenMP; its main justification is compatibility with systems where work must be
split across MPI ranks.}

\subsection{Hybrid MPI+OpenMP vs.\ MPI+Pthreads}

Comparing the two hybrid variants at matched $P\times T$ layouts isolates the
effect of the intra node threading model: the outer MPI structure, the row
partition, and the collective calls are identical in both. The MPI+OpenMP
variant is faster wherever the inner team has more than one thread
(4$\times$2: 1.7631\,s vs.\ 1.7807\,s; 2$\times$4: 3.9960\,s vs.\ 4.0245\,s;
1$\times$8: 7.8006\,s vs.\ 8.0027\,s), mirroring what is already seen at the
single technology level between pure OpenMP (6.26$\times$) and pure Pthreads
(5.74$\times$). The gap comes from the same two mechanisms: (i)~OpenMP's
\code{schedule(dynamic,4)} smooths the triangular Phase~3 imbalance that the
pthreads \code{work\_range} static partition cannot, and (ii)~OpenMP reuses
one thread team across phases, whereas the pthreads harness pays
\code{pthread\_create}/\code{pthread\_join} overhead three times per run. At
$T{=}1$, where the threading model is effectively bypassed and only the MPI
layer matters, the two hybrids converge. In fact, Hybrid PT 8$\times$1
(1.7702\,s) edges Hybrid 8$\times$1 (1.7970\,s) by $\approx$1.5\%, well within
run to run noise, because there is no OpenMP runtime to amortise and the
pthreads binary skips the OMP team creation cost entirely.

\subsection{Amdahl's Law}\label{sec:amdahl}

Amdahl's Law explains why the CPU implementations remain below the ideal
eight worker limit. The similarity and prediction phases are highly parallel,
but the measured speedups also include scheduling overhead, memory bandwidth
pressure, MPI collectives, and lower all core turbo frequency. OpenMP therefore
comes closest to the ideal, Pthreads loses additional efficiency through static
work splitting and thread creation, and MPI loses more to collective
communication.

\section{Conclusions}\label{sec:conclusions}
% ══════════════════════════════════════════════════════════════════════════════

\begin{enumerate}
  \item \textbf{All versions preserve correctness.} MAE and RMSE are computed
        against the serial prediction array. Every implementation achieves
        MAE $=\textbf{0.0000}$ and RMSE $=\textbf{0.0000}$, and the
        similarity-matrix checksum is bit-identical at $\textbf{942.387323}$.
  \item \textbf{OpenMP achieves the best CPU performance.} $\times$6.26 speedup at
        8 threads, 78\% efficiency. Dynamic scheduling and thread reuse explain
        the advantage.
  \item \textbf{Pthreads is competitive but less efficient.} $\times$5.74 at 8T,
        72\% efficiency. Static row assignment and per phase thread creation
        limit scalability.
  \item \textbf{MPI achieves good speedup with communication overhead.}
        $\times$5.10 at 8P, 64\% efficiency. On this single node benchmark,
        \code{MPI\_Allgatherv} communication keeps it behind OpenMP.
  \item \textbf{Hybrid performance is best when the inner thread count is small
        on a single node.} The four MPI+OpenMP configurations rank
        4$\times$2 ($\times$4.84) $\gtrsim$ 8$\times$1 ($\times$4.75)
        $\gg$ 2$\times$4 ($\times$2.14) $>$ 1$\times$8 ($\times$1.09); the
        MPI+Pthreads variant ranks 8$\times$1 ($\times$4.82) $\gtrsim$
        4$\times$2 ($\times$4.80) $\gg$ 2$\times$4 ($\times$2.12) $>$
        1$\times$8 ($\times$1.07). The two hybrids agree to within $\approx$2\%
        at every layout. The OMP variant wins at $T>1$ (dynamic scheduling),
        while the PT variant wins at $T=1$ (no OMP runtime overhead). On this
        single node run, both hybrids stay behind pure OpenMP.
  \item \textbf{CUDA provides the largest speedup by far.} Benchmarked on the
        NVIDIA RTX 3050, the GPU completes similarity + prediction in
        \textbf{0.2202\,s}, a \textbf{38.78$\times$} speedup over the serial
        baseline and $\approx$6$\times$ faster than the best 8 thread CPU run.
        The similarity kernel launches a grid with 1{,}016{,}064 logical
        threads.
\end{enumerate}

\textbf{Final ranking (single node):}
\begin{center}
CUDA GPU ($\times$38.78)
$>$ OpenMP 8T ($\times$6.26)
$>$ Pthreads 8T ($\times$5.74)
$>$ MPI 8P ($\times$5.10) \\[2pt]
$>$ Hybrid 4$\times$2 ($\times$4.84)
$>$ Hybrid PT 8$\times$1 ($\times$4.82)
$>$ Hybrid PT 4$\times$2 ($\times$4.80)
$>$ Hybrid 8$\times$1 ($\times$4.75) \\[2pt]
$>$ Hybrid 2$\times$4 ($\times$2.14)
$>$ Hybrid PT 2$\times$4 ($\times$2.12)
$>$ Hybrid 1$\times$8 ($\times$1.09)
$>$ Hybrid PT 1$\times$8 ($\times$1.07)
\end{center}
